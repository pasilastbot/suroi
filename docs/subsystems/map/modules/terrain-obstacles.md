## Purpose

Defines how the map's terrain floors, obstacles, buildings, loot, and named locations are spatially placed during map generation. This module covers collision validation, placement strategies by spawnMode, and the systems that prevent overlaps.

---

## Map Structure

### Dimensions & Boundaries

Each `MapDefinition` specifies:

```typescript
readonly width: number        // World units (e.g., 1632 for normal map)
readonly height: number       // World units (e.g., 1632 for normal map)
readonly oceanSize: number    // Border width in units (e.g., 128)
readonly beachSize: number    // Walkable sand strip width (e.g., 32)
```

**Layout (concentric bounds):**

```
┌─────────────────────────────────────────────┐
│                  Ocean                      │
│  (non-walkable water, oceanSize units)      │
│ ┌───────────────────────────────────────┐   │
│ │  Beach (sand/walkable, beachSize)     │   │
│ │ ┌─────────────────────────────────┐   │   │
│ │ │  Island (grass/trees/buildings) │   │   │
│ │ │  (where players spawn/fight)     │   │   │
│ │ └─────────────────────────────────┘   │   │
│ └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

Three derived hitboxes:

| Hitbox | Type | Purpose |
|--------|------|---------|
| `oceanHitbox` | (implicit) | Ocean border; collision check in `getRandomPosition()` |
| `beachHitbox` | `GroupHitbox<RectangleHitbox[]>` | 4 beach strips (top, bottom, left, right) |
| `islandHitbox` | `RectangleHitbox` | Walkable landmass interior |

### Jagged Coast Polygon

The beach/island boundary is **not a rectangle** — it's a `PolygonHitbox` with jagged edges to break up the geometry visually.

Generated by `jaggedRectangle(rect, spacing=16, variation, seededRandom)`:

- Walks the four edges at **16-unit intervals**
- Offset each vertex by ±`variation/2` in the perpendicular direction
- `variation = min(8, 2 * oceanSize / beachSize)` — fewer vertices = larger variation
- Uses `SeededRandom` so client and server produce **identical polygons** from the same seed

**Impact on placement:**
- Terrain floors use `beachHitbox` and `grassHitbox` (the two jagged polygons)
- Obstacles/buildings use `getRandomPosition()` with `spawnMode`, which checks these polygon collisions

## Terrain System

### `Terrain` Class (`common/src/utils/terrain.ts`)

Core spatial structure for ground floors and river lookups:

```typescript
export class Terrain {
    readonly width: number;           // Map width in units
    readonly height: number;          // Map height in units
    readonly cellSize = 64;           // Grid cell size (world units per cell)
    readonly floors: Map<Hitbox, { floorType: FloorNames; layer: Layer | number }>;
    readonly rivers: readonly River[];
    readonly waterType: FloorNames;   // Default water floor type (e.g., "water", "ice", "sand")
    readonly beachHitbox: PolygonHitbox;
    readonly grassHitbox: PolygonHitbox;
    readonly groundRect: RectangleHitbox;
}
```

**Cost Model:**
- Internal grid: cells = `floor(width / 64) × floor(height / 64)`
- Each cell stores which `rivers[]` and `floors{}` overlap it
- Lookup: `terrain.getRiversInHitbox(hitbox)` returns only rivers that spatially overlap

**Construction:**

1. **Allocate grid** — `_grid[x][y]` for each 64×64 cell
2. **Generate coast** — `jaggedRectangle()` creates `beachHitbox` and `grassHitbox`
3. **Seed floor types** — interior gets default grass; building interiors override with wood/stone
4. **Index rivers** — for each `River`, find all cells whose 64×64 boundary overlaps `river.bankHitbox`, add river to `_grid[cell]`

### `River` Class

```typescript
export class River {
    readonly width: number;
    readonly points: readonly Vector[];   // Spline waypoints (60 for rivers, 25 for trails)
    readonly waterHitbox: Hitbox | undefined;   // Only for rivers (not trails)
    readonly bankHitbox: Hitbox;
    readonly isTrail: boolean;

    /** Get random point inside this river's water area */
    getRandomPosition(onTrails: boolean, radius: number): Vector;

    /** Get point on spline at parameter t ∈ [0.0, 1.0] */
    getPosition(t: number): Vector;

    /** Get velocity vector (tangent) at parameter t */
    getTangent(t: number): Vector;
}
```

**Key properties:**

- **waterHitbox** — only present for rivers (not trails). Used to check if objects are "in a river".
- **bankHitbox** — visible collision boundary (used for `River` or `Trail` spawn mode checks)
- **getRandomPosition()** — picks a random point on the river centerline (via spline sampling)

## Collision Validation & Spawn Position Finding

### `getRandomPosition()` Algorithm

The core placement function that enforces **all** collision rules:

```typescript
// @file server/src/map.ts:995
getRandomPosition(
    initialHitbox: Hitbox,
    params?: {
        getPosition?: () => Vector;
        collides?: (position: Vector) => boolean;
        spawnMode?: MapObjectSpawnMode;
        spawnOffset?: Vector | { min: Vector; max: Vector };
        spawnRadius?: number;
        bunkerHitbox?: Hitbox;
        scale?: number;
        orientation?: Orientation;
        maxAttempts?: number;
        orientationConsumer?: (orientation: Orientation) => void;
        ignoreClearings?: boolean;
        river?: River;
    }
): Vector | undefined {
    // ...
}
```

**Algorithm (pseudo-code):**

```
maxAttempts = 200 (default)
for attempt in 0..maxAttempts {
  position = params.getPosition()  // Random position for spawnMode
  
  // Check custom collider
  if params.collides(position) → continue
  
  // Check collision with grid objects (obstacles, buildings)
  hitbox = initialHitbox.transform(position, scale, orientation)
  for object in game.grid.intersectsHitbox(hitbox) {
    if hitbox.collidesWith(object.hitbox) → continue
  }
  
  // Check terrain-specific constraints (based on spawnMode)
  switch spawnMode {
    case Grass / GrassAndSand / Beach / Ring:
      // Check not in river (except trails for GrassAndSand)
      for river in terrain.getRiversInHitbox(hitbox) {
        if riverBank.isPointInside(position) → continue
        if spawnMode == GrassAndSand and !river.isTrail and hitbox.collidesWith(river.water) → continue
      }
      // Check not in clearing
      if !params.ignoreClearings {
        for clearing in this.clearings {
          if clearing.collidesWith(hitbox) → continue
        }
      }
    
    case River:
      // Must be entirely inside river water
      for all corners of hitbox:
        if !river.waterHitbox.isPointInside(corner) → continue
    
    case Trail:
      // Must not be in river water (trails don't have water)
      if this.isInRiver(hitbox) → continue
  }
  
  // Success
  return position
}

return undefined  // Failed to find position
```

**Key collision layers:**

1. **Grid objects** — obstacles and buildings on the same layer (checked via spatial index `game.grid.intersectsHitbox()`)
2. **Rivers/Trails** — `river.bankHitbox` and `river.waterHitbox` (checked via `terrain.getRiversInHitbox()`)
3. **Clearings** — `RectangleHitbox[]` stored in `this.clearings[]` (can be skipped with `ignoreClearings: true`)
4. **Terrain polygons** — jagged coast is implicitly checked in spawn mode logic (e.g., Beach spawn mode uses `beachHitbox`)

### `MapObjectSpawnMode` Strategies

Defined in [common/src/constants.ts](common/src/constants.ts#L205):

```typescript
export enum MapObjectSpawnMode {
    Grass,            // Island grass only (no beach)
    GrassAndSand,     // Island grass AND beach sand
    River,            // Only inside river water hitbox
    Beach,            // Only on beach strips
    Trail,            // On trail centerline (not in water)
    Ring              // Circular ring around map center (e.g., for spawns)
}
```

**Implementation in getRandomPosition():**

| Mode | Position Generator | Constraints |
|------|-----|---|
| `Grass` | `randomVector(interior bounds without beach)` | Not in river banks or water; not in clearing |
| `GrassAndSand` | `randomVector(interior bounds including beach)` | Not in rivers (except trails are OK); not in clearing |
| `River` | `river.getRandomPosition(false)` picks point on river centerline | All 4 corners must be in `river.waterHitbox` |
| `Beach` | `randomPoint()` from oriented beach strip | Not in river; not in ocean; not in clearing |
| `Trail` | `river.getRandomPosition(true)` for trails only | Must not be in any river's waterHitbox |
| `Ring` | `Vec.add(center, Vec.fromPolar(randomRotation(), spawnRadius))` | No terrain constraint (for player spawns) |

**Example usage:**

- **Obstacles** — typically `Grass` or `GrassAndSand` depending on definition
- **River obstacles** — `River` or `Trail` spawnMode
- **Loot** — `GrassAndSand` (can be on beaches)
- **Buildings** — `Grass` (default), or other modes per `BuildingDefinition.spawnMode`

## Building & Room Generation

### Building Placement Strategy

Defined in `_generateBuildings(idString, count)` @ [server/src/map.ts:470](server/src/map.ts#L470):

**For normal buildings:**

1. Try up to **100 attempts** to find a valid position
2. For each attempt:
   - Resolve building definition
   - Randomize `orientation` based on `BuildingDefinition.rotationMode`
   - Call `getRandomPosition(spawnHitbox, { orientation, spawnMode, maxAttempts: 400 })`
   - If found, check **quadrant constraints** (below)
3. If position is valid per quadrant rules, call `generateBuilding(buildingDef, position, orientation)`

**For bridge buildings:**

- Skips normal grid placement
- Iterates `terrain.rivers` instead
- For each river, tries fractional spline positions `t ∈ [0.1-0.4] ∪ [0.6-0.9]` (in random order)
- Picks orientation with smallest angular difference from river tangent
- Checks `spawnHitbox` doesn't collide with existing objects or beach

**Building Definition Fields:**

```typescript
// @file common/src/definitions/buildings.ts (conceptual)

readonly bunkerSpawnHitbox?: Hitbox;    // Basement layer hitbox
readonly rotationMode: RotationMode;    // Full / Limited / Binary / None
readonly spawnMode?: MapObjectSpawnMode;  // Default: Grass
readonly spawnRadius?: number;          // For Ring mode
readonly spawnOffset?: Vector;          // Offset from spawn point
readonly spawnOrientation?: Orientation;  // Override random orientation

readonly bridgeHitbox?: Hitbox;         // Markers this as a bridge
readonly bridgeMinRiverWidth?: number;
readonly bridgeSpawnRanges?: [[number, number], [number, number]];  // Default: [[0.1, 0.4], [0.6, 0.9]]
```

### Quadrant Constraints

Map is divided into 4 quadrants @ midpoint `(width/2, height/2)`:

```
┌─────────────┬─────────────┐
│   Quad 1    │   Quad 2    │
│  (x<mid,    │  (x>=mid,   │
│   y<mid)    │   y<mid)    │
├─────────────┼─────────────┤
│   Quad 3    │   Quad 4    │
│  (x<mid,    │  (x>=mid,   │
│   y>=mid)   │   y>=mid)   │
└─────────────┴─────────────┘
```

**Rules:**

- **Major buildings** (those in `MapDefinition.majorBuildings`):
  - Max 1 per quadrant
  - Any two major buildings must be ≥ 150,000 units² apart (distance²)
- **Quad-limited buildings** (those with `MapDefinition.quadBuildingLimit[idString]`):
  - Each quadrant can have at most `quadBuildingLimit[idString]` of that type

**Tracked state:**

```typescript
private readonly quadBuildings: Record<1 | 2 | 3 | 4, string[]> = { 1: [], 2: [], 3: [], 4: [] };
private readonly quadMajorBuildings: Array<1 | 2 | 3 | 4> = [];
private readonly majorBuildingPositions: Vector[] = [];
```

When a building spawns, it instantiates defined obstacles and sub-buildings:

**Relative positioning:** Each building obstacle is defined with:

- `.locked` / `.activated` — initial state

## Obstacle Placement

### Standalone Obstacle Generation

```typescript
// @file server/src/map.ts:807
private _generateObstacles(
    definition: ReifiableDef<ObstacleDefinition>,
    count: number,
    getPosition?: () => Vector   // Optional custom position generator
): void {
    const def = Obstacles.reify(definition);
    count = this.mapScale(count);
    
    for (let i = 0; i < count; i++) {
        const scale = randomFloat(def.scale?.spawnMin ?? 1, def.scale?.spawnMax ?? 1);
        const variation = random(0, def.variations - 1) as Variation;
        const rotation = GameMap.getRandomRotation(def.rotationMode);
        
        const position = this.getRandomPosition(def.spawnHitbox ?? def.hitbox, {
            getPosition,
            scale,
            orientation: def.rotationMode === RotationMode.Limited ? rotation : undefined,
            spawnMode: def.spawnMode,
            ignoreClearings: clearingDef?.allowedObstacles.includes(def.idString) ?? false
        });
        
        if (position) this.generateObstacle(def, position, { layer: Layer.Ground, scale, variation });
    }
}
```

**Key features:**

- Count is scaled by `mapScale()` when `scale !== 1`
- Each obstacle gets a random `scale` from `[spawnMin, spawnMax]`
- Rotation mode determines if orientation is passed (affects collision shape)
- `ignoreClearings` is **inverted**: if the obstacle is in `clearingDef.allowedObstacles`, it CAN spawn in clearings

### Obstacle Clump Generation

Groups of obstacles placed in circular/arc patterns:

```typescript
// @file server/src/map.ts:930
private _generateObstacleClumps(clumpDef: ObstacleClump): void {
    const clumpAmount = this.mapScale(clumpDef.clumpAmount);
    const { obstacles, minAmount, maxAmount, radius, jitter } = clumpDef.clump;
    
    for (let i = 0; i < clumpAmount; i++) {
        // Find clump center
        const centerPosition = this.getRandomPosition(
            new CircleHitbox(radius + jitter),
            { spawnMode: Obstacles.reify(obstacles[0]).spawnMode }
        );
        if (!centerPosition) continue;
        
        // Spread obstacles around center
        const amountOfObstacles = this.mapScale(random(minAmount, maxAmount));
        const offset = randomRotation();
        const step = τ / amountOfObstacles;
        
        for (let j = 0; j < amountOfObstacles; j++) {
            this.generateObstacle(
                pickRandomInArray(obstacles),
                Vec.add(
                    randomPointInsideCircle(centerPosition, jitter),
                    Vec.fromPolar(j * step + offset, radius)
                )
            );
        }
    }
}
```

**Layout:**

```
Clump center at (centerX, centerY):

           jitter region (random offset ±jitter)
                    ↓
radius  ↗ ┌────────────────┐
          │  ○ ○ ○ ○ ○     │  ← obstacle positions
          │ ○ ○ ○ ○ ○ ○    │     at radial distance `radius`
          │┌────────┐      │     with jitter scatter
          ││ center ││      │     in circular arc pattern
          │└────────┘      │
          │  ○ ○ ○ ○ ○     │
          └────────────────┘
```

**Example (normal map):**

```typescript
obstacleClumps: [
    {
        clumpAmount: 100,           // 100 clusters
        clump: {
            obstacles: ["oak_tree"],
            minAmount: 2,
            maxAmount: 3,           // 2–3 trees per cluster
            radius: 12,             // 12 units apart
            jitter: 5               // ±5 unit scatter
        }
    },
    // birch and pine clusters...
]
```

### Loot Generation

```typescript
// @file server/src/map.ts:965
private _generateLoots(table: string, count: number): void {
    for (let i = 0; i < count; i++) {
        const loot = getLootFromTable(this.game.modeName, table);
        
        const position = this.getRandomPosition(
            new CircleHitbox(5),
            { spawnMode: MapObjectSpawnMode.GrassAndSand }
        );
        
        if (!position) {
            this.game.warn(`Failed to find valid position for loot from '${table}'`);
            continue;
        }
        
        for (const item of loot) {
            this.game.addLoot(
                item.idString,
                position,
                Layer.Ground,
                { count: item.count, jitterSpawn: false }
            );
        }
    }
}
```

**Key constraints:**

- Spawn mode is `GrassAndSand` — can be on beach
- Small hitbox (5-unit radius) — allows dense placement
- Each loot position yields **multiple items** from the table (e.g., ammo + gun)
- `jitterSpawn: false` — all items at exact same position (unlike player drops which scatter)

```typescript
// @file server/src/map.ts:214
if (mapDef.places) {
    packet.places = mapDef.places.map(({ name, position }) => {
```

## Biome & Tileset Selection

### Floor Type Switching by Mode

Different game modes can override the default water floor type:


**Mode examples:**

| Mode | `replaceWaterBy` | Effect |
|------|---------|---------|
| `normal` | `"water"` | Standard blue water |
| `winter` | `"ice"` | Slippery ice instead of water |
| `fall` | `"water"` | Standard water (autumn theme) |
| `halloween` | `"water"` | Standard water (Halloween theme) |

### Visual Biome Indicators

While **server logic** doesn't distinguish "biomes" (it just uses floor types and building lists), the **client-side rendering** uses:

1. **Terrain grid** — grass, sand, water, ice tiles rendered from the `FloorNames`
2. **Obstacle variants** — some obstacles have mode-specific HDR sprites:
   ```typescript
   // @file server/src/map.ts:636
   if (obstacleData.outdoors && this.game.mode.obstacleVariants) {
       const _modeName = Modes[this.game.modeName].similarTo ?? this.game.modeName;
       idString = `${idString}_${_modeName}`;  // e.g., "oak_tree_fall"
   }
   ```
3. **Building visuals** — same approach; each mode can have themed buildings

### Map-Specific Biome Definition

The "biome" is defined by the combination of:

| Component | Determined By |
|-----------|---|
| Water floor type | `ModeName` → `Modes[modeName].replaceWaterBy` |
| Tree/obstacle types | `MapDefinition.obstacleClumps[]` and `obstacles{}` |
| Building list | `MapDefinition.majorBuildings[]`, `buildings{}`, `bridges[]` |
| River/trail config | `MapDefinition.{rivers, trails}` |
| Named locations | `MapDefinition.places[]` |

**Example:** Winter map biome = winter water (ice) + winter obstacle sprites + winter buildings + winter places.

## River & Trail Obstacle Placement

### Density-Based Spawning

```typescript
// @file server/src/map.ts:386
private _generateRiverObstacles(riverDef: RiverDefinition, onTrails: boolean): void {
    for (const river of this.terrain.rivers) {
        if (onTrails !== river.isTrail) continue;
        
        for (const obstacle in riverDef.obstacles) {
            const amount = riverDef.obstacles[obstacle] * river.width * river.points.length / 500;
            
            const definition = Obstacles.reify(obstacle);
            const hitbox = definition.spawnHitbox ?? definition.hitbox;
            
            for (let i = 0; i < amount; i++) {
                const position = this.getRandomPosition(hitbox, {
                    getPosition: () => river.getRandomPosition(
                        definition.spawnMode === MapObjectSpawnMode.Trail,
                        hitbox.type === HitboxType.Circle ? hitbox.radius : 0
                    ),
                    spawnMode: definition.spawnMode,
                    ignoreClearings: true,
                    river
                });
                
                if (position) this.generateObstacle(definition, position);
            }
        }
    }
}
```

**Density formula:**

```
amount = obstacleRecord[idString] × river.width × river.points.length ÷ 500
```

**Why divide by 500?**
- Allows definition authors to use human-scale integers (e.g., `16`) instead of decimals (e.g., `0.032`)
- For a typical river (width=20, 60 points): `16 × 20 × 60 ÷ 500 ≈ 38.4` obstacles

**Example (normal map):**

```typescript
rivers: {
    minAmount: 1,
    maxAmount: 2,
    minWidth: 12,
    maxWidth: 18,
    obstacles: {
        river_rock: 16,     // ~40 rocks per 60-point river
        lily_pad: 6         // ~15 pads per 60-point river
    }
}
```

## Clearings System

### Clearing Definition

```typescript
readonly clearings?: {
    readonly minWidth: number;
    readonly minHeight: number;
    readonly maxWidth: number;
    readonly maxHeight: number;
    readonly count: number;
    readonly allowedObstacles: readonly ReferenceTo<ObstacleDefinition>[];
    readonly obstacles: ReadonlyArray<{
        readonly idString: ReferenceTo<ObstacleDefinition>;
        readonly min: number;
        readonly max: number;
    }>;
};
```

### Generation

```typescript
// @file server/src/map.ts:423
private _generateClearings(clearingDef: MapDefinition["clearings"]): void {
    if (!clearingDef) return;
    
    const count = this.mapScale(clearingDef.count);
    
    for (let i = 0; i < count; i++) {
        const width = this.mapScale(randomFloat(clearingDef.minWidth, clearingDef.maxWidth));
        const height = this.mapScale(randomFloat(clearingDef.minHeight, clearingDef.maxHeight));
        let hitbox = RectangleHitbox.fromRect(width, height);
        
        let validPositionFound = false;
        for (let attempts = 0; attempts < 100; attempts++) {
            if ((position = this.getRandomPosition(hitbox)) !== undefined) {
                validPositionFound = true;
                this.clearings.push(hitbox = RectangleHitbox.fromRect(width, height, position));
                break;
            }
        }
        
        if (!validPositionFound) {
            this.game.warn("Failed to find valid position for clearing");
            continue;
        }
        
        // Populate clearing with defined obstacles
        for (const obstacle of clearingDef.obstacles) {
            this._generateObstacles(
                obstacle.idString,
                random(obstacle.min, obstacle.max),
                () => hitbox.randomPoint()
            );
        }
    }
}
```

**Key behaviors:**

- Clearings are **non-colliding `RectangleHitbox`** stored in `this.clearings[]`
- Used as collision check in `getRandomPosition()` — most spawn modes reject positions inside clearings (unless `ignoreClearings: true`)
- Obstacles **within** a clearing are generated with `ignoreClearings: true` and use `clearingHitbox.randomPoint()` as position generator
- Example use case: fall map has 2 clearings with logs/pebbles/rocks

## Performance Considerations

### Spatial Indexing

**Collision checks:** Instead of iterating **all** map objects, placement uses `game.grid.intersectsHitbox(hitbox)`:

```typescript
// @file server/src/map.ts:1083-1090 (pseudo)
for (const object of this.game.grid.intersectsHitbox(hitbox)) {
    let objectHitbox: Hitbox;
    if ("spawnHitbox" in object) {
        objectHitbox = object.spawnHitbox;
    } else if (object.hitbox) {
        objectHitbox = object.hitbox;
    } else {
        continue;
    }
    
    if (collidableObjects[object.type] && hitbox.collidesWith(objectHitbox)) {
        collided = true;
        break;
    }
}
```

Grid cells are **32 units** (from `GameConstants.gridCellSize`). Lookup is O(k) where k is grid cells overlapping the query hitbox.

**Terrain grid:** Rivers are indexed by 64×64 cell:

```typescript
// @file common/src/utils/terrain.ts (conceptual)
readonly cellSize = 64;
_grid: Record<"x,y", River[]>;

getRiversInHitbox(hitbox): River[] {
    // Find all cells overlapping hitbox, collect rivers
}
```

### Procedural Generation Budget

Generation happens **once per game** in the `GameMap` constructor:

- Typical map: ~1632×1632 units
- ~200 buildings, ~500 obstacles, ~100 loot piles, 2 rivers
- Runtime: **< 1 second** on modern hardware
- Bottleneck: `maxAttempts` retry loops for building placement (400 attempts max per building)

### Caching

The serialized `MapPacket` is computed **once** and stored in `this.buffer: ArrayBuffer`:

```typescript
const stream = new PacketStream(new ArrayBuffer(1 << 16));  // 64 KiB
stream.serialize(packet);
this.buffer = stream.getBuffer();
```

Every joining player receives the **same buffer** — no re-serialization.

## Known Issues & Gotchas

### 1. **Major Building Quadrant Checks Use Distance², Not Euclidean Distance**

```typescript
// @file server/src/map.ts:523 (buggy)
this.majorBuildingPositions.some(pos => Geometry.distanceSquared(pos, position) < 150000)
```

**Issue:** Uses `distanceSquared` (150,000 units²) instead of `distance < sqrt(150,000)`.
- Intended minimum distance: ~387 units
- Actual minimum: varies (square boundary instead of circle)

**Impact:** Major buildings can be closer than intended in diagonal patterns.

### 2. **Spawn Position Attempts Don't Account for Scaled Maps**

When `scale > 1`, counts are scaled by `mapScale()`, but `maxAttempts` in `getRandomPosition()` is **not scaled**:

```typescript
const maxAttempts = params?.maxAttempts ?? 200;  // Fixed, not scaled
```

**Impact:** On very large scaled maps, placement may fail for large objects (buildings) due to insufficient retry budget.

**Mitigation:** The placement loop in `_generateBuildings` has its own `attempts < 100` check before calling `getRandomPosition`, so very large buildings eventually log a warning.

### 3. **River Obstacle Amount Is Non-Integer**

```typescript
const amount = riverDef.obstacles[obstacle] * river.width * river.points.length / 500;
for (let i = 0; i < amount; i++)  // Loops with non-integer amount!
```

**Issue:** `amount` can be `38.4`, and the loop will only execute 38 times (truncated).

**Impact:** Actual river obstacle counts vary by 1–2 below expectations.

**Fix:** Use `Math.round()` or Poisson sampling.

### 4. **Bridges Don't Respect Building Quad Limits**

Bridge placement bypasses `quadBuildingLimit` checks:

```typescript
// @file server/src/map.ts:556-608
// Bridge generation doesn't check quadBuildingLimit!
```

**Impact:** Bridges are not limited per quadrant, even if defined in `quadBuildingLimit`.

**Use case:** This is intentional for gameplay balance (rivers cross quadrants naturally), but inconsistent.

### 5. **Clearings Can Trap Spawn Positions**

If clearings are placed too densely, spawning may fail:

```typescript
// If few clearings fail to place, spawn fails for later clearings
// No retry or alternative placement strategy
```

**Impact:** On fall/Halloween maps with multiple clearings, if placement fails early, later clearings are not attempted.

### 6. **Unsorted Spawn Modes Don't Have Consistent Bias**

`randomVector()` uses uniform distribution, but river/trail spawn modes use `river.getRandomPosition()`:

```typescript
// Grass/Beach spawn modes: uniform random
// River/Trail spawn modes: sampled from spline (non-uniform due to point clusters)
```

**Impact:** River obstacles may cluster around river waypoint bunches.

### 7. **Overlap Zones Between Rivers & Trails**

If both rivers and trails are defined, their collision zones can overlap:

```typescript
rivers.push(...this._generateRivers(mapDef.trails, seededRandom, true));
rivers.push(...this._generateRivers(mapDef.rivers, seededRandom));
// Both stored in same rivers[] array
```

**Impact:** Obstacle placement logic checks `river.isTrail` to distinguish, but visually they can overlap.
